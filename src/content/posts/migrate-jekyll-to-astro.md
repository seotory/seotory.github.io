이 블로그는 2016년에 `jekyll`로 만들었다. 그리고 10년 동안 글만 몇 개 올렸을 뿐, 블로그 자체에는 거의 손을 대지 않았다. 잘 돌아가는 것처럼 보였기 때문이다. 정적 사이트라 서버가 죽을 일이 없고, `GitHub Pages`가 알아서 빌드해 주니 신경 쓸 일이 없었다. 그런데 오랜만에 열어 보니 상태가 처참했다. `jekyll`은 4.2.2에 머물러 있었고, `jQuery` 3.1.1은 2016년에 받아 온 그대로였으며, 아이콘은 `FontAwesome` 킷, 구글 애널리틱스는 2023년 7월에 종료된 UA(`analytics.js`)였다.

마지막 항목이 특히 뼈아팠다. 애널리틱스가 종료된 지 3년이 지났으니 그동안 수집된 데이터가 하나도 없었다는 뜻이다. 방문자 수를 확인해 본 적이 없어서 그 사실조차 몰랐다. 정적 사이트가 안 죽는다는 말은 틀린 것이 아니라 부정확하다. **사이트 자체는 죽지 않지만, 사이트가 의존하는 것들은 조용히 죽는다.** 그리고 정적 사이트는 그것이 죽어도 여전히 200을 응답하기 때문에, 방치하는 한 알아차릴 방법이 없다.

이왕 손대는 김에 `astro`로 갈아엎기로 했다. 이 글은 그 이전 과정의 기록이며, 다루는 내용은 다음과 같다.

- 옮기기 전에 죽은 런타임 의존성부터 걷어낸 판단
- `jekyll`이 프레임워크로 주던 것을 `astro`에서 직접 떠안는 지점 — 카테고리와 URL
- 글과 URL을 그대로 유지한 채 옮기는 방법
- 증상은 비슷하지만 원인이 각기 달랐던 이전 후의 문제들

# 옮기기 전에: 죽은 런타임 의존성부터 청산

프레임워크를 바꾸기 전에, 먼저 `jekyll` 상태에서 라이브러리부터 정리했다. 이전 작업과 정리 작업을 한꺼번에 하면 무언가 깨졌을 때 원인이 프레임워크 전환인지 라이브러리 교체인지 분리할 수 없기 때문이다. 순서를 나누는 것 자체가 이후 디버깅 비용을 줄이는 결정이었다.

## jQuery가 실제로 하던 일

`jQuery` 3.1.1이 CDN에서 로드되고 있었다. 무엇에 쓰고 있는지 코드를 열어 보니, 전부 합쳐 두 가지였다. 모바일 햄버거 메뉴를 토글하는 것, 그리고 검색창에서 엔터를 누르면 구글 검색으로 보내는 것. **30KB짜리 라이브러리를 받아서 하는 일이 클래스 토글과 이벤트 리스너 하나였던 셈이다.** 2016년이라면 브라우저 호환성을 고려한 정당한 선택이었겠지만, `querySelector`와 `classList`가 어디서나 동작하는 지금은 근거가 사라진 의존성이다.

```js
document.querySelector('.navbar-burger').addEventListener('click', (e) => {
  e.currentTarget.classList.toggle('is-active');
});
```

버전을 최신으로 올리는 선택지도 있었지만, 어차피 제거할 대상이라면 올릴 이유가 없었다. 그대로 걷어냈다.

## FontAwesome이 실제로 하던 일

`FontAwesome`은 킷 방식으로 들어와 있었다. 이 방식은 JS를 받고, 그 JS가 CSS를 받고, 그 CSS가 다시 폰트 파일을 받는 3단 로딩 구조다. 이렇게 해서 실제로 쓰고 있던 아이콘은 홈, 검색, 정보 세 개였다. **아이콘 세 개를 위해 폰트 파일과 런타임 의존성을 통째로 지고 있었다.**

교체 대상은 `Lucide`의 SVG를 인라인으로 박는 방식으로 정했다. 네트워크 요청이 0이 되고, 렌더링이 더 선명하며, 색은 `currentColor`로 문맥을 따라온다. 재사용은 컴포넌트 하나로 해결된다. 여기까지 하고 나니 외부 런타임 의존성이 전부 사라졌고, 이 상태를 기준선으로 삼아 `astro`로 넘어갔다.

# jekyll과 astro의 구조적 차이

`astro`로 옮기며 가장 오래 헤맨 지점은 문법이나 빌드가 아니었다. `jekyll`이 프레임워크 차원에서 알아서 해 주던 것들이, `astro`에는 아예 개념 자체가 없다는 사실이었다.

## 카테고리는 프레임워크가 주지 않는다

`jekyll`에서 카테고리는 프레임워크가 주는 것이었다. `_posts/dev/java/` 폴더에 글을 넣으면 카테고리가 자동으로 잡히고, `site.categories`로 어디서든 꺼내 쓸 수 있었다. 내가 한 일은 폴더를 만든 것뿐이다.

`astro`에는 그런 것이 없다. 카테고리도, 태그도, 시리즈도 프레임워크가 알지 못한다. 문서를 못 찾는 줄 알고 한참을 뒤졌지만 정말로 없는 것이 맞았다. 대신 있는 것은 Content Collections와 스키마다.

```ts
// src/content.config.ts
const posts = defineCollection({
  loader: glob({ pattern: '**/*.md', base: './src/content/posts' }),
  schema: z.object({
    title: z.string(),
    date: z.coerce.date(),
    category: z.enum(CATEGORY_PATHS),
    tags: z.array(z.string()).default([]),
  }),
});
```

즉 `astro`의 입장은 분류 체계를 직접 설계하라는 것이다. `category`라는 필드는 프레임워크가 아는 개념이 아니라 내가 스키마에 적어서 생긴 필드일 뿐이고, 그것으로 목록을 만들고 URL을 뚫는 일도 전부 내 몫이다.

처음에는 블로그 프레임워크가 카테고리를 주지 않는다는 것이 불친절하게 느껴졌다. 그런데 쓰다 보니 이쪽이 낫다는 판단으로 바뀌었다. `z.enum(CATEGORY_PATHS)`처럼 허용된 값을 못 박아 두면 오타가 빌드 단계에서 걸리기 때문이다.

```text
category: Invalid input: expected "dev"
```

`jekyll` 시절에는 `categories: devv`라고 오타를 내도 아무 일도 일어나지 않았다. 글 하나짜리 유령 카테고리가 조용히 생길 뿐이었다. 실제로 이 블로그에는 글은 있는데 메뉴에는 없는 카테고리와, 메뉴에는 있는데 글이 0개인 카테고리가 둘 다 몇 년째 존재하고 있었고, 옮기는 과정에서야 발견했다. **프레임워크가 알아서 해 준다는 말은, 뒤집으면 내가 틀려도 알려 주지 않는다는 뜻이다.**

## URL은 설정이 아니라 파일 위치가 결정한다

`jekyll`은 `_config.yml`에 URL 규칙을 적었다.

```yml
permalink: /post/:title
paginate_path: "/list/:num/"
```

`astro`에는 이런 설정이 아예 없다. 찾을 필요도 없다. 파일을 어디에 두느냐가 곧 URL이다.

```text
src/pages/index.astro                  →  /
src/pages/post/[slug].astro            →  /post/<slug>/
src/pages/list/[page].astro            →  /list/<n>/
src/pages/categories/[...path].astro   →  /categories/<a/b/c>/
```

파일명에 대괄호가 붙으면 틀만 있고 실제 값은 코드가 정한다는 뜻이다. `[slug]`는 한 칸을, `[...path]`는 슬래시를 포함한 여러 칸을 받는다. 실제 값은 `getStaticPaths()`가 만들어 낸다.

```js
export const getStaticPaths = (async () => {
  const posts = await getPublishedPosts();
  return posts.map((post) => ({
    params: { slug: post.id },
    props: { post },
  }));
});
```

이 함수가 배열을 하나 반환하면 빌드 시점에 그 개수만큼 HTML이 찍혀 나온다. 페이지를 미리 전부 생성해 두는 방식이라 런타임 라우팅이랄 것이 없다.

여기서 `post.id`가 파일명이라는 점이 이전 작업의 핵심이었다. 덕분에 `2017-09-03-java-builder-pattern.markdown`을 `java-builder-pattern.md`로 이름만 바꾸면 `/post/java-builder-pattern/`이 그대로 나온다. `jekyll`이 파일명에서 날짜를 떼고 `:title`을 뽑아 쓰던 것과 결과가 같다. 날짜는 front matter에 있으니 파일명에 다시 있을 이유가 없다. **`permalink`를 바꿨을 때 어떤 URL이 나올지 머릿속으로 시뮬레이션할 필요가 없고, 파일 경로를 보면 URL이 그대로 보인다.**

## 페이징도 파일 하나로 끝난다

파일이 곧 URL이라면 글이 늘어날 때마다 `/list/4/`, `/list/5/`를 직접 만들어야 하는가 하는 의문이 든다. 그렇지 않다. 대괄호 파일 하나면 된다.

```js
export const getStaticPaths = (async ({ paginate }) => {
  const posts = await getPublishedPosts();
  return paginate(posts, { pageSize: 10 });
});
```

`paginate()`가 글 개수를 보고 페이지 수를 계산해 경로까지 만들어 준다. 반신반의하며 글을 21개에서 33개로 늘려 빌드해 보니, 파일을 건드리지 않아도 `/list/4/`가 생겼다. 정확히는 글을 쓸 때 페이지가 생기는 것이 아니라, 빌드할 때마다 전체를 다시 계산해서 찍는 것이다.

# 옮기는 과정

## 글: 44개 중 실제 공개분은 21개

`_posts` 아래에 마크다운이 44개 있었는데, 그중 실제로 공개된 것은 21개였다. 나머지 23개는 `published: false`, 즉 초안이었다. 여기에 더해 `.hide/` 폴더에 제목만 잡아 둔 파일이 18개 더 있었다. **정리하면 글 21개를 위해 파일 62개를 이고 다녔던 것이고, 이 숫자 역시 이번에 세어 보고 처음 알았다.**

옮기며 손댄 작업은 다음 셋이다.

- 파일명에서 날짜 접두사 제거 — 앞서 설명한, 파일명이 곧 slug가 되는 구조 때문이다.
- `kramdown`의 IAL 문법 제거 — `[링크](url){:target="_blank"}` 같은 표기다. `astro`가 쓰는 마크다운 파서는 이 문법을 모르는데, 문제는 처리가 안 되는 것이 아니라 `{:target="_blank"}`가 화면에 글자 그대로 찍힌다는 점이다.
- `{% raw %}` 래퍼 제거 — Liquid 템플릿 엔진에게 이 안은 건드리지 말라고 알리는 태그인데, Liquid가 사라졌으니 존재 이유도 사라졌다.

뒤의 둘은 결국 Liquid의 흔적이다. `jekyll`은 마크다운 파일을 템플릿으로도 취급해서, 본문에 `{{ }}`가 있으면 그것을 변수로 해석하려 든다. `astro`는 마크다운을 그냥 마크다운으로 읽는다.

## 코드 하이라이팅: rouge에서 shiki로

`rouge`가 `shiki`로 바뀐다. `astro`에 내장되어 있어 설정 한 줄이면 된다.

```js
markdown: {
  shikiConfig: { theme: 'github-light' },
}
```

`rouge`는 CSS 클래스를 붙여 두고 색은 별도 스타일시트에서 입히는 방식이었는데, `shiki`는 빌드 시점에 VS Code의 문법 분석기를 돌려 색을 인라인 스타일로 직접 박아 넣는다. 그래서 테마 CSS를 따로 관리할 필요가 없다. 대신 **언어를 지정하지 않은 코드블록은 색이 입혀지지 않는다.** 색을 박아 넣을 근거가 없으니 당연한 동작이고 `rouge`도 마찬가지였지만, 이 사소한 차이가 뒤에서 한 건의 장애로 돌아온다.

# CSS 프레임워크 교체: Bulma에서 Tailwind로

`Bulma`도 결국 걷어냈다. 처음에는 1.0으로 올려 계속 쓰려 했지만, 커스텀이 필요할 때마다 `!important`를 붙이게 되는 것이 문제였다. **프레임워크를 이기는 수단이 CSS 특이도 싸움밖에 없다면, 그것은 프레임워크를 쓰는 것이 아니라 프레임워크와 싸우는 것이다.**

교체 대상은 `tailwind`로 정했다. v4부터는 설정이 JS 파일이 아니라 CSS로 들어간다.

```css
@import "tailwindcss";

@theme {
  --color-accent-500: hsl(244 55% 55%);
  --font-sans: 'Pretendard Variable', Pretendard, sans-serif;
  --radius-card: 1rem;
}
```

여기서 스스로 정한 원칙이 하나 있다. 되도록 프레임워크의 기능으로 해결하고, 커스텀 CSS를 새로 짜지 않는다는 것이다. `@theme`에 토큰을 등록하면 `bg-accent-500`, `rounded-card` 같은 유틸리티가 자동으로 생긴다. 색이 필요할 때 값을 어딘가에 하드코딩하는 대신 토큰을 하나 늘리는 식이고, 이렇게 해 두면 나중에 액센트 색을 바꿀 때 고칠 곳이 한 군데로 모인다. **설정을 프레임워크의 어휘로 표현하면, 변경도 프레임워크의 어휘 안에서 끝난다.**

CSS 번들은 최종적으로 37KB, gzip 기준 7KB가 나왔다. `Bulma`는 CDN에서 프레임워크 전체를 받아왔으므로 — 쓰지 않는 컴포넌트까지 포함해서 — 이 정도 차이는 예상된 결과다.

# 카테고리 정리: 합치되 2뎁스는 살려 둔다

옮기며 카테고리를 세어 보니 `dev/java`, `dev/javascript`, `dev/python` 식으로 잘게 나뉘어 있었는데, 글 21개가 전부 `dev/*` 아래에 있었다. 다른 1뎁스 카테고리는 글이 0개였다. **분류 대상의 100%가 한 갈래에 몰려 있으면 그 분류는 아무것도 분류하지 못한다.** 전부 `dev` 하나로 합쳤다.

다만 여기서 한 가지를 정했다. 2뎁스 카테고리는 언제든 다시 생길 수 있으니, 지금 안 쓴다고 UI와 CSS를 지우지는 않는다. 드롭다운 코드를 삭제해 버리면 나중에 카테고리를 하나 추가할 때 UI를 다시 만들어야 하기 때문이다. 그래서 데이터 구조와 렌더링은 2뎁스를 그대로 지원하되, 지금은 데이터가 1뎁스일 뿐인 상태로 두었다.

```ts
export const CATEGORY_NAV: NavItem[] = [{ title: 'Dev', path: 'dev', use: true }];
```

`items`를 달아 주면 그 즉시 드롭다운이 생긴다. 카테고리 목록 페이지도 `[...path]`라서 슬래시가 몇 개든 받는다. **이 결정은 뒤의 리다이렉트 처리에서 함정이 되어 돌아오는데, 그 이야기는 뒤에서 다룬다.**

# 배포: Pages는 astro를 빌드하지 않는다

`GitHub Pages`는 `jekyll`만 자동으로 빌드해 준다. `Pages`가 원래 `jekyll` 호스팅으로 출발한 서비스라서 그렇다. `astro`는 `GitHub Actions`로 직접 빌드해 결과물을 올려야 한다.

```yml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: withastro/action@v3
        with:
          node-version: 22
  deploy:
    needs: build
    steps:
      - uses: actions/deploy-pages@v4
```

여기서 놓치기 쉬운 것이 하나 있다. 저장소 설정에서 Pages 소스를 바꿔야 한다는 점이다. `Settings > Pages > Source`가 `Deploy from a branch`로 되어 있으면, 워크플로가 아무리 초록불이어도 `Pages`는 여전히 브랜치를 바라본다. **빌드는 성공하는데 사이트는 그대로인 상태**가 되는데, 실패하는 것보다 이쪽이 더 헷갈린다. 성공 로그가 문제 없음을 알려 주는 듯 보이기 때문이다. 소스를 `GitHub Actions`로 바꾸면 해결된다.

```bash
$ gh api -X POST repos/<user>/<repo>/pages -f build_type=workflow
```

# 이전 후에 만난 문제들

이제 이 글의 본론이다. 옮기고 나서 여러 문제를 만났는데, 상당수는 증상만 보고는 원인이 어디인지 알 수 없는 종류였다. 이런 장애에서 먼저 할 일은 원인을 추측하는 것이 아니라 **증상을 만들어 낸 주체가 어디인지 확인하는 것**이다. 그 주체가 문제의 계층을 알려 주기 때문이다.

## node 버전: 공식 액션의 기본값이 본체 요구와 어긋난다

첫 배포가 바로 터졌다.

```text
Node.js v20.20.2 is not supported by Astro!
Please upgrade Node.js to a supported version: ">=22.12.0"
```

`withastro/action`은 `astro` 팀이 만든 공식 액션인데, 기본값이 `node` 20이다. 그리고 `astro`는 22.12 이상을 요구한다. **공식 액션의 기본값이 본체의 요구사항과 맞지 않는 것이다.** 로컬에서는 22를 쓰고 있어 빌드가 잘 됐고, 그래서 CI에서만 터졌다. 위 워크플로에 `node-version: 22`가 명시된 이유다. 액션이 언젠가 고칠 문제지만, 명시해 두면 기본값이 또 바뀌어도 흔들리지 않는다.

## 다크모드: 메이저 버전은 가만히 있어도 기본 동작을 바꾼다

`Bulma`를 0.9.3에서 1.0.4로 올렸더니 블로그가 시커멓게 변했다. CSS는 한 줄도 건드리지 않은 상태였다.

원인은 `Bulma` 1.0부터 `prefers-color-scheme`을 자동으로 따라간다는 데 있었다. 0.9까지는 그런 개념이 없어 항상 흰 바탕이었다. 즉 버그가 아니라 새 기능이고, OS를 다크모드로 쓰는 사람에게만 나타난다. 나는 맥을 다크모드로 쓰고 있었으니 나에게만 시커멓게 보였던 것이다. **메이저 버전을 올린다는 것은 내가 아무것도 하지 않아도 기본 동작이 바뀔 수 있다는 뜻**임을 새삼 확인한 사례였다. 흰 바탕을 고정하려면 테마를 명시하면 된다.

```html
<html data-theme="light">
```

## 사라진 애드센스: 코드가 아니라 죽은 apex 도메인

이전 후에 만난 문제 중 원인이 가장 멀리 있었던 건이다.

언젠가부터 광고가 나오지 않고 있었다. 당연히 코드부터 의심했지만, 파 보니 전부 멀쩡했다. 로더 스크립트도 붙고, `<ins>` 태그도 제자리에 있고, `push()`도 호출되고, `ads.txt`도 200으로 응답했다. 요청은 구글까지 도달했는데, 구글이 `data-ad-status="unfilled"`로 답하고 있었다. 줄 광고가 없다는 뜻이다. **요청이 정상적으로 나가고 거절만 돌아온다면, 문제는 코드가 아니라 계정 쪽이다.** 여기서 조사 방향을 코드에서 콘솔로 옮겼다.

애드센스 콘솔의 사이트 목록에는 `blog.seotory.com`이 아니라 apex 도메인인 `seotory.com`이 등록되어 있었고, 상태가 이랬다.

```text
승인 상태: 주의 필요
상태 세부정보: 사이트 소유권을 확인할 수 없음, 사이트가 다운되었거나 사용할 수 없음
Ads.txt 상태: 찾을 수 없음
```

루트 도메인이 죽어 있었다. DNS를 확인하니 폐기된 옛 `GitHub Pages` IP를 가리키고 있었다.

```bash
$ dig +short A seotory.com
192.30.252.153      # 폐기된 옛날 GitHub Pages IP
```

`GitHub Pages`가 몇 년 전 IP 대역을 바꿨는데, 나는 `blog` 서브도메인만 옮기고 apex는 방치했던 것이다. `http`로는 404, `https`는 연결 자체가 실패했다. apex로 들어오는 사람이 없어 그동안 몰랐다.

문제는 이 죽음이 블로그까지 끌고 들어갔다는 점이다. 애드센스는 등록된 `seotory.com`을 확인하러 갔다가 사이트가 죽어 있으니 소유권 확인에 실패했고, 그 결과 서브도메인 `blog`을 포함한 도메인 전체에 광고를 끊었다. **블로그 코드는 처음부터 무관했다.** 그렇다면 `blog.seotory.com`을 새로 등록하면 되지 않나 싶었지만, 그것도 막혔다.

```text
URL은 올바른 최상위 도메인이어야 합니다.
```

애드센스는 apex 도메인만 등록할 수 있고, 서브도메인은 apex에 딸려가는 구조다. 결국 `seotory.com`을 살리는 것 외에 방법이 없었다. apex 하나를 위해 블로그를 통째로 옮길 수도, 리다이렉트 하나를 위해 서버를 띄울 수도 없었는데, DNS가 `cloudflare`에 있어 호스팅 없이 해결됐다.

- apex `A` 레코드를 `192.0.2.1`로 바꾸고 프록시를 켠다. 이 IP는 문서용으로 예약되어 실제로 라우팅되지 않지만, 프록시를 켜면 `cloudflare`가 앞단에서 요청을 처리하므로 뒤에 서버가 없어도 무방하다. 요컨대 없는 서버를 가리키는 척하는 A 레코드다.
- `Page Rules`로 `seotory.com/*`를 `https://blog.seotory.com/$1`로 301 리다이렉트한다.

여기서 `$1`이 핵심이다. 이것이 없으면 모든 요청이 홈으로 향해 `seotory.com/ads.txt`가 블로그의 `ads.txt`에 도달하지 못한다. ads.txt 규격은 리다이렉트를 허용하므로, 경로만 보존되면 구글이 따라와서 읽는다.

```bash
$ curl -sI https://seotory.com/ads.txt | grep -i location
location: https://blog.seotory.com/ads.txt
```

이 상태에서 애드센스의 Ads.txt 스니펫 확인을 눌렀더니 상태가 승인됨으로 바뀌었다. **블로그 코드는 끝까지 한 줄도 고치지 않았다.** 이 사건이 남긴 원칙은 명확하다. 광고가 나오지 않을 때 코드를 먼저 뒤지면 안 된다. 브라우저가 요청을 보내고 응답을 받고 있다면, 코드는 이미 용의선상에서 빠진다.

## Typography의 기본값 두 가지: 백틱과 어두운 코드블록

마크다운 본문은 `@tailwindcss/typography`의 `prose` 클래스가 처리한다. `<h2>`, `<p>`, `<ul>` 하나하나에 스타일을 주는 대신 부모에 `prose` 하나만 붙이면 되니 편한데, 기본값 두 가지가 문제를 일으켰다.

첫째, 인라인 코드에 백틱을 붙인다. 화면에 백틱 문자가 그대로 보이길래 마크다운이 깨진 줄 알았지만, HTML을 열어 보니 멀쩡한 `<code>`였다. 범인은 `prose`였고, CSS로 백틱을 그려 넣고 있었다.

```css
.prose :where(code)::before { content: "`" }
.prose :where(code)::after  { content: "`" }
```

인라인 코드를 배경색 없이도 구분되게 하려는 의도지만, 나는 배경색을 주었으니 백틱은 중복이다. `content: none`으로 끄면 된다.

둘째, 코드블록이 어둡다고 가정한다. `prose`의 기본 코드블록은 어두운 배경에 밝은 글자다. 나는 사이트가 밝으니 배경만 밝게 바꿨는데, 그 결과 언어를 지정하지 않은 코드블록만 글자가 밝은 채로 남아 흰 배경에 흰 글자가 됐다. 언어를 지정한 블록은 멀쩡했다. `shiki`가 인라인 스타일로 색을 박아 넣어 `prose`의 글자색을 밀어냈기 때문이다. **그래서 특정 블록만 안 보이는 현상**이 나온다. 앞서 하이라이팅 절에서 말한 그 차이가 여기서 터진 것이다. 배경과 글자색을 함께 지정하면 된다.

```css
--tw-prose-pre-bg: #f6f8fa;
--tw-prose-pre-code: #24292e;
```

그런데 이 변수를 `.prose-blog`에 넣었더니 이번에는 모든 코드블록이 어두워졌다. 고치려다 더 망가진 것이다. 원인은 특이도였다. 플러그인의 규칙은 `.prose`로 특이도 (0,1,0)이고, 내 `.prose-blog`도 (0,1,0)이다. 같은 값이면 나중에 선언된 쪽이 이기는데, 빌드 순서상 플러그인이 뒤였다. 클래스를 하나 더 붙여 (0,2,0)으로 올리니 해결됐다.

```css
.prose.prose-blog { --tw-prose-pre-bg: #f6f8fa; }
```

유틸리티 프레임워크를 쓰면 특이도 싸움에서 벗어난다는 통념이 있지만, 플러그인이 끼면 그렇지도 않다.

## 리다이렉트가 부활한 카테고리를 가로챈다

카테고리를 `dev` 하나로 합치면서 옛 URL 19개가 사라지게 됐다. 검색엔진에 색인된 링크라 404로 버릴 수 없어 `astro`의 `redirects`로 넘겼다.

```js
redirects: {
  '/categories/dev/java': '/categories/dev/',
  // ...
}
```

여기에 함정이 있다. 앞서 2뎁스 카테고리는 언제든 다시 생길 수 있다고 정해 두었는데, 나중에 `dev/java`를 되살리면 이 리다이렉트가 실제 카테고리 페이지를 조용히 가로챈다. 경고도 에러도 없다. 실험 삼아 되살려 보니 정말로 글 목록 대신 리다이렉트 스텁이 나왔다. 이것은 언젠가 반드시 밟을 지뢰였다.

그래서 리다이렉트 목록을 손으로 관리하는 대신, 현재 살아 있는 카테고리는 목록에서 자동으로 빠지도록 만들었다.

```js
const categoryRedirects = Object.fromEntries(
  RETIRED_CATEGORY_PATHS
    .filter((p) => !CATEGORY_PATHS.includes(p))   // 부활하면 자동으로 빠진다
    .map((p) => [`/categories/${p}`, '/categories/dev/']),
);
```

`CATEGORY_PATHS`는 메뉴 정의에서 뽑아낸 값이라, 카테고리를 되살리는 순간 리다이렉트에서 저절로 빠진다. **미래의 내가 이 예외를 기억할 것이라고 믿는 대신, 구조가 기억하게 만드는 편이 낫다.**

한 가지 덧붙이면, `GitHub Pages`는 정적 호스팅이라 진짜 301을 반환하지 못한다. HTTP 헤더를 만들려면 서버가 필요한데 없기 때문이다. `astro`의 `redirects`는 대신 `meta refresh`와 `canonical`이 담긴 HTML을 찍어 준다. 브라우저는 잘 따라가고 검색엔진도 `canonical`을 존중하지만, 엄밀한 301은 아니다. 제대로 된 301이 필요하면 애드센스 때와 마찬가지로 `cloudflare`에서 처리해야 한다.

## 몇 년째 적용되지 않던 한글 폰트

이것은 `jekyll` 시절부터 있던 문제인데 이제야 발견했다. `Bulma`의 기본 폰트 스택에는 한글 글꼴이 하나도 없었다.

```text
Inter, "SF Pro", "Segoe UI", Roboto, ... , sans-serif
```

폰트 스택은 글자 하나하나마다 앞에서부터 훑어 그 글자를 가진 첫 폰트를 쓰는 방식이다. 그러니 한글은 목록 전체를 통과해 맨 끝 `sans-serif`로 떨어지고, 결국 OS가 고른 폰트로 렌더된다. 맥은 Apple SD Gothic Neo, 윈도우는 맑은 고딕. **즉 영문은 Inter, 한글은 보는 사람마다 다른 폰트로 나오고 있었다.** 글의 90%가 한글인 블로그가 몇 년째 이 상태였다. 눈치채지 못한 이유는 단순하다. 한글이 깨지지 않고 잘 보였기 때문이다. 폰트가 아예 없으면 두부(□)라도 떠서 티가 나는데, 폴백은 조용히 잘 동작한다.

`_sass`를 열어 보니 더 흥미로웠다. `NanumBarunGothic`을 `@font-face`로 정의는 해 두었는데 어디에도 적용하지 않고 있었다. `font-family`에 넣는 것을 잊은 채, 폰트 파일만 몇 년째 저장소에 얹고 다닌 것이다. `Pretendard`로 정리했다. 라틴과 한글을 한 폰트로 덮어 섞일 일이 없고, 라틴 메트릭이 Inter와 호환이라 영문 인상도 크게 바뀌지 않는다.

```css
--font-sans: 'Pretendard Variable', Pretendard, -apple-system, sans-serif;
```

한글 폰트는 글자 수가 많아 파일이 무거운 것이 약점인데, dynamic subset 빌드를 쓰면 유니코드 영역을 90여 개로 쪼개 실제로 사용된 글자가 든 조각만 내려받는다.

# 정리

이전 전후의 구성을 비교하면 다음과 같다.

| 항목 | 전 | 후 |
| --- | --- | --- |
| 프레임워크 | jekyll 4.2.2 (ruby) | astro 7 (node) |
| CSS | Bulma CDN (전체) | tailwind 37KB / gzip 7KB |
| JS | jQuery 3.1.1 | 없음 |
| 아이콘 | FontAwesome 킷 | 인라인 SVG |
| 코드 하이라이팅 | rouge | shiki |
| 배포 | Pages 자동 빌드 | GitHub Actions |
| 애널리틱스 | UA (죽어 있음) | GA4 |
| 한글 폰트 | 미지정 | Pretendard |

URL은 `/post/:title`, `/list/N/`, `/categories/`, `/feed/`까지 전부 그대로 유지했다. 색인된 링크를 하나도 버리지 않은 것이 개인적으로는 가장 만족스러운 결과다.

이번 이전의 결론을 정리하면 다음과 같다.

- 프레임워크가 알아서 해 주는 편리함에는 대가가 있다. `jekyll`은 카테고리와 URL을 자동으로 처리해 주었지만, 그 대가로 오타 난 카테고리도, 적용 안 된 폰트도, 죽은 애널리틱스도 몇 년간 알려 주지 않았다. `astro`는 스키마에 적은 대로만 동작하고 틀리면 빌드가 깨진다. 귀찮지만 이쪽이 안전하다.
- 증상만으로 원인을 추측하기 전에, 그 증상을 만들어 낸 주체가 어디인지부터 확인한다. 애드센스의 `unfilled` 응답은 코드가 아니라 계정과 도메인을 가리키고 있었다.
- 프레임워크의 기능으로 표현할 수 있는 설정은 커스텀 코드로 덮어쓰지 않는다. `!important`나 특이도 싸움이 시작되면, 그것은 프레임워크를 쓰는 것이 아니라 이기려 드는 신호다.
- 미래의 예외는 사람의 기억이 아니라 구조가 감당하게 한다. 부활한 카테고리가 리다이렉트에서 자동으로 빠지도록 만든 것이 그 예다.

그리고 이전 과정에서 만난 문제의 절반은 프레임워크와 무관했다. 죽어 있던 apex 도메인, 정의만 하고 적용하지 않은 폰트 같은 것들이다. 이것들은 `astro`가 고쳐 준 것이 아니라, **오랜만에 블로그를 열어 보았기 때문에** 드러난 것이다. 아직 방치 중인 `jekyll` 블로그가 있다면, 한 번쯤 열어 볼 만하다. 생각보다 많은 것이 조용히 죽어 있을 것이다.
